# Trinity/OpenSpec 整合上下文

> 此文件作为 Agent Orchestrator 和 Trinity/OpenSpec 之间的桥接层
> Agent 启动时会自动读取此文件获取工作上下文

## 启动协议 (MANDATORY - 每次启动必须执行)

### Phase 1: 获取活跃变更

```bash
# 读取活跃变更名
ACTIVE=$(cat openspec/.active 2>/dev/null || echo "")

if [ -z "$ACTIVE" ]; then
  echo "⚠️  没有活跃变更。"
  echo "请先运行以下命令之一："
  echo "  /trinity:new <change-name>    # 创建新变更"
  echo "  /trinity:ff <change-name>     # 快速创建所有 artifacts"
else
  echo "📍 活跃变更: $ACTIVE"
fi
```

### Phase 2: 读取变更 Artifacts

```bash
if [ -n "$ACTIVE" ]; then
  CHANGE_DIR="openspec/changes/$ACTIVE"
  
  echo "📋 读取任务列表..."
  if [ -f "$CHANGE_DIR/tasks.md" ]; then
    cat "$CHANGE_DIR/tasks.md"
  else
    echo "⚠️  tasks.md 不存在，请先运行 /trinity:continue"
  fi
  
  echo ""
  echo "📐 读取设计文档..."
  if [ -f "$CHANGE_DIR/design.md" ]; then
    cat "$CHANGE_DIR/design.md"
  fi
  
  echo ""
  echo "📏 读取规格文件..."
  if [ -d "$CHANGE_DIR/specs" ]; then
    find "$CHANGE_DIR/specs" -name "*.md" -exec cat {} \;
  fi
  
  echo ""
  echo "📄 读取提案..."
  if [ -f "$CHANGE_DIR/proposal.md" ]; then
    cat "$CHANGE_DIR/proposal.md"
  fi
fi
```

### Phase 3: 读取追踪状态

```bash
echo "📊 项目级追踪状态..."

if [ -f "task_plan.md" ]; then
  echo "=== task_plan.md ==="
  cat task_plan.md
fi

if [ -f "findings.md" ]; then
  echo ""
  echo "=== findings.md (技术决策) ==="
  cat findings.md
fi

if [ -f "progress.md" ]; then
  echo ""
  echo "=== progress.md (进度日志) ==="
  cat progress.md
fi

# 变更级追踪
if [ -n "$ACTIVE" ] && [ -d "$CHANGE_DIR/.trinity" ]; then
  echo ""
  echo "📂 变更级追踪..."
  
  if [ -f "$CHANGE_DIR/.trinity/stage.md" ]; then
    cat "$CHANGE_DIR/.trinity/stage.md"
  fi
  
  if [ -f "$CHANGE_DIR/.trinity/task-plan.md" ]; then
    cat "$CHANGE_DIR/.trinity/task-plan.md"
  fi
  
  if [ -f "$CHANGE_DIR/.trinity/progress.md" ]; then
    cat "$CHANGE_DIR/.trinity/progress.md"
  fi
fi
```

## 工作流指南

### Trinity Skills (优先使用)

| Skill | 用途 | 触发时机 |
|-------|------|---------|
| `/trinity:new <name>` | 创建新变更 | 开始新功能 |
| `/trinity:ff <name>` | 快速创建所有 artifacts | 快速原型 |
| `/trinity:continue` | 继续下一个 artifact | 分阶段推进 |
| `/trinity:apply` | 执行 tasks.md | 开始编码 |
| `/trinity:verify` | 验证实现 | 质量检查 |
| `/trinity:archive` | 归档变更 | 完成功能 |

**使用示例**:
```
/trinity:new snake-core
/trinity:ff snake-core
/trinity:apply
/trinity:verify
/trinity:archive
```

### Superpowers Skills (底层方法论)

| Skill | 用途 | 触发时机 |
|-------|------|---------|
| `superpowers:brainstorming` | 需求分析 | 不清楚做什么 |
| `superpowers:writing-plans` | 编写计划 | 复杂任务开始前 |
| `superpowers:test-driven-development` | TDD 工作流 | 实现功能 |
| `superpowers:systematic-debugging` | 系统化调试 | 遇到 Bug |
| `superpowers:verification-before-completion` | 完成验证 | 提交前 |
| `superpowers:receiving-code-review` | 处理 Review | 收到反馈 |

**TDD 循环**:
1. **RED**: 先写失败测试
2. **GREEN**: 写最少代码通过
3. **REFACTOR**: 清理代码

## 3-Strike 协议

遇到错误时按以下流程处理：

| 尝试 | 动作 | 记录到 | 说明 |
|------|------|--------|------|
| Attempt 1 | 诊断并修复 | `progress.md` | 分析错误日志，找到根因 |
| Attempt 2 | 尝试替代方案 | `findings.md` | 记录为什么第一个方案不行 |
| Attempt 3 | 重新思考问题 | `findings.md` | 可能理解错了问题 |
| 3次失败 | 升级给用户 | `task_plan.md [BLOCKED]` | 标记为阻塞，等待人工 |

## 项目约定

- **语言**: 中文（简体）
- **技术栈**: TypeScript, Vite, Canvas
- **任务粒度**: 每个任务 2-5 分钟可完成
- **规格格式**: Given/When/Then (Gherkin)
- **提交格式**: Conventional Commits (feat:, fix:, chore:)

## Agent Orchestrator 整合

### CI 失败自动处理

当 CI 失败时，Agent Orchestrator 会自动发送：

```
CI 失败。按照 3-Strike 协议处理：
- Attempt 1: 诊断并修复，记录到 progress.md
- Attempt 2: 尝试替代方案，记录到 findings.md
- Attempt 3: 重新思考，记录到 findings.md
使用 superpowers:systematic-debugging 找根因。
```

### Review Comments 自动处理

当收到 Review 请求变更时：

```
Review 请求变更。使用 superpowers:receiving-code-review 处理反馈。
每个修改都要更新 findings.md 记录决策。
```

## 模块边界 (多Agent并行开发)

### Agent 分配

| Agent | 角色 | CLI | 模块 | Worktree |
|-------|------|-----|------|----------|
| Agent 1 | 核心开发者 | Claude Code | `src/core/*` | `~/.worktrees/snake-game-core/` |
| Agent 2 | 渲染开发者 | OpenCode | `src/renderer/*` | `~/.worktrees/snake-game-render/` |
| Agent 3 | 输入开发者 | Claude Code | `src/input/*` | `~/.worktrees/snake-game-input/` |

### 共享接口

所有 Agent 必须遵循 `src/core/types.ts` 中定义的接口契约。

### 合并策略

1. Core 模块先合并到 main
2. Renderer 和 Input 模块基于最新 main 开发
3. 集成测试在所有模块合并后执行
